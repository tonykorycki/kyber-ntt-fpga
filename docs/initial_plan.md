## 1. IP Definition

### What the IP Does

This project implements a hardware accelerator for negacyclic polynomial multiplication in the ring $\mathbb{Z}_q[x]/(x^n + 1)$, the core arithmetic primitive of the CRYSTALS-Kyber post-quantum key encapsulation mechanism (NIST PQC Standard, FIPS 203 / ML-KEM).

Classical public-key cryptography (RSA, ECDH) is vulnerable to quantum computers via Shor's algorithm. Kyber is a lattice-based scheme believed to be quantum-resistant. Its performance bottleneck is repeated polynomial multiplication in $\mathbb{Z}_{3329}[x]/(x^{256}+1)$. A single Kyber key exchange requires on the order of 20–30 such multiplications; each is $O(n^2)$ naively but $O(n \log n)$ via the Number Theoretic Transform (NTT).

The IP accelerates this bottleneck on the FPGA fabric (PL), offloading the computation from the ARM Cortex-A9 (PS). The PS runs the rest of the Kyber protocol in C++ and calls the IP for every polynomial multiplication via memory-mapped AXI-Lite registers, receiving results back through shared BRAM.

The initial implementation targets small, simulation-friendly parameters ($n=4$, $q=17$) to enable rapid development and verification. The architecture is parameterized so that it scales to full Kyber parameters ($n=256$, $q=3329$) by regenerating the twiddle ROM and updating two constants.

### PS–PL Interaction

```
PS (ARM Cortex-A9)                        PL (FPGA Fabric)
──────────────────                        ────────────────
Write a[0..n-1] → BRAM A        ──>
Write b[0..n-1] → BRAM B        ──>
Write START → AXI-Lite reg      ──>       ntt_top HLS IP
                                           ├── pre_twist on a, b
                                           ├── ntt_forward on a, b
                                           ├── pointwise_mul
                                           ├── ntt_inverse
                                           └── post_twist -> BRAM C
Poll DONE ← AXI-Lite reg        <──
Read c[0..n-1] ← BRAM C         <──
```

**Data transferred:**
- Input: two coefficient vectors `a[0..n-1]` and `b[0..n-1]`, each coefficient a 12-bit unsigned integer ($q = 3329 < 2^{12}$)
- Output: one coefficient vector `c[0..n-1]`, same format
- Control: START (write) and DONE (read) registers, auto-generated by HLS via `#pragma HLS INTERFACE s_axilite`

**Interface:**
- AXI-Lite slave: control/status registers (START, DONE) — generated by HLS, not hand-written
- Dual-port BRAM: bulk data transfer of coefficient arrays, arbitrated by `bram_mux.sv`

### Mathematical Operations

The algorithm computes $c(x) = a(x) \cdot b(x) \bmod (x^n + 1, q)$ using the negacyclic NTT. Full derivation in `docs/mathematic_derivation.md`. Summary:

**Pre-twist** (eq. 7):
$$\tilde{a}_i = a_i \cdot \psi^i \bmod q$$

**Forward NTT — Cooley-Tukey butterfly** (eq. CT), $\log_2 n$ stages:
$$\hat{a}_k = E_k + \omega_m^j \cdot O_k, \qquad \hat{a}_{k+m/2} = E_k - \omega_m^j \cdot O_k$$

**Pointwise multiply** (eq. 17):
$$C_k = A_k \cdot B_k \bmod q$$

**Inverse NTT — Gentleman-Sande butterfly** (eq. GS), $\log_2 n$ stages, scale by $n^{-1}$:
$$a[k+j] = u + t, \qquad a[k+j+m/2] = \omega_m^{-j} \cdot (u - t)$$

**Post-twist** (eq. 9):
$$c_i = \tilde{c}_i \cdot \psi^{-i} \bmod q$$

The key hardware operation at every step is modular multiplication mod $q$, implemented via Barrett reduction (eq. 20). Twiddle factor tables ($\psi^i$, $\omega^i$, and their inverses) are precomputed by `scripts/gen_twiddle_rom.py` and stored in on-chip BRAM.

## 2. IP Architecture

### Design Boundary Principle

The architecture follows a strict three-layer boundary:

- **HLS (C++):** all arithmetic compute and AXI-Lite control registers. HLS generates the register interface automatically — it is never hand-written in RTL.
- **SystemVerilog:** structural wiring only. Instantiates HLS IP blocks and BRAMs, arbitrates BRAM port ownership per pipeline stage. No `always` blocks except in `bram_mux.sv`.
- **C++ PS driver:** memory-mapped I/O to HLS-generated AXI-Lite registers, BRAM DMA, and the DONE polling loop. The only file that knows the AXI-Lite register map.

### Sub-Module Responsibilities

#### `barrett.cpp`
Standalone Barrett modular multiplier: given $a$, $b$, $q$, returns $(a \cdot b) \bmod q$. Factored out so it can be independently verified and reused by all four compute stages. Verified exhaustively in `tb_barrett.cpp` against Python's `%` operator.

#### `ntt_engine.cpp`
Cooley-Tukey forward NTT and Gentleman-Sande inverse NTT. Controlled by a flag selecting $\omega$ vs $\omega^{-1}$ and enabling the $n^{-1}$ post-scale. HLS `#pragma HLS PIPELINE` and `#pragma HLS UNROLL` directives applied to the butterfly inner loop. Twiddle factors read from BRAM; coefficient array passed by reference.

#### `twist.cpp`
Pre-twist ($\tilde{a}_i = a_i \cdot \psi^i$) and post-twist ($c_i = \tilde{c}_i \cdot \psi^{-i}$). $n$ independent modular multiplications — trivially parallelizable with `#pragma HLS UNROLL`.

#### `pointwise.cpp`
Element-wise $C_k = A_k \cdot B_k \bmod q$. $n$ independent Barrett reductions — trivially parallelizable.

#### `ntt_top.cpp`
Top-level HLS function. Sequences the pipeline stages and owns all AXI-Lite registers via `#pragma HLS INTERFACE s_axilite`. Register map is generated by HLS synthesis and consumed directly by `ps/ntt_driver.cpp` — never hardcoded separately.

#### `bram_mux.sv`
The only RTL file with logic. Arbitrates dual-port BRAM access between the HLS IP and the PS AXI interconnect based on pipeline stage. All other `.sv` files are purely structural.

#### `ntt_driver.cpp`
Memory-mapped I/O to the HLS AXI-Lite register map. Exposes a single function: `ntt_mul(a, b, c)` — writes inputs to BRAM, pulses START, polls DONE, reads output from BRAM. The register base address and offsets come directly from the HLS synthesis report.

### Data Flow

```
BRAM_A (a) ──> pre_twist ──> ntt_forward ──┐
                                            ├──> pointwise_mul ──> ntt_inverse ──> post_twist ──> BRAM_C (c)
BRAM_B (b) ──> pre_twist ──> ntt_forward ──┘
```

All inter-stage data stays on-chip. The PS only touches BRAM_A, BRAM_B (writes), and BRAM_C (read) — never intermediate buffers.

### Parameterization

| Parameter  | Dev ($n=4$)  | Full Kyber ($n=256$) |
|------------|-------------|----------------------|
| `n`        | 4           | 256                  |
| `q`        | 17          | 3329                 |
| `COEF_WIDTH` | 5 bits    | 12 bits              |
| `LOG2_N`   | 2           | 8                    |

Switching parameters requires only re-running `gen_twiddle_rom.py` and updating `COEF_WIDTH` and `LOG2_N`. The pipeline structure is otherwise identical.

## 3. Verification Strategy

Each sub-module is verified independently against the Python golden model before integration. The Python golden model (`golden/ntt.py`) is itself verified against two independent references: exhaustive Barrett checking against Python's `%`, and `ntt_mul` vs `schoolbook_nwc`.

| Module | Test method | Reference |
|---|---|---|
| `barrett` | Exhaustive all $(a,b) \in \mathbb{Z}_q^2$ | Python `(a*b) % q` |
| `ntt_engine` | Random inputs | `naive_ntt` (direct eq. 1 evaluation) |
| `twist` | Roundtrip pre then post = identity | Python `pre_twist` / `post_twist` |
| `pointwise` | Random inputs | Python `pointwise_mul` |
| `ntt_top` | Random polynomial pairs | Python `ntt_mul` vs `schoolbook_nwc` |
| `ntt_top_wrapper` | SV sim with AXI-Lite transactions | `sim/vectors/test_vectors.txt` |

Test vectors for the SV testbench are generated by `golden/test_ntt.py:generate_vectors()` and written to `golden/test_vectors.txt`.

## 4. Extended Goal: Kyber Wrapper

Once the NTT multiplication IP is verified, `ps/main.cpp` will be extended to implement the full Kyber KEM protocol (key generation, encapsulation, decapsulation), calling `ntt_driver.cpp` for every polynomial multiply. This demonstrates end-to-end post-quantum key exchange with hardware-accelerated inner loops and allows a direct software-only vs hardware-accelerated latency comparison.
